	  collapsed:: true

		- 靠「找相似」的模型
			- | 模型 | 「找相似」的具体逻辑 | 为什么不用梯度下降？ |
			  | ---- | ---- | ---- |
			  | [[KNN]] | 预测时，找和当前样本**距离最近的 K 个邻居**，用邻居的标签投票 / 均值做预测 | 没有需要训练的参数（K 是超参数，不是模型参数），不需要迭代优化，预测时直接算距离 |
			  | [[K-Means]] | 迭代计算样本到簇中心的**距离**，把相似样本分到同一个簇，更新簇中心 | 没有需要学习的参数，只靠距离计算分组，全程无梯度、无损失函数 |
			  | **推荐系统（协同过滤）** | 找和你**兴趣相似的用户**，把相似用户喜欢的物品推荐给你 | 核心是计算用户 / 物品的相似度（用距离度量），不需要训练参数 |
			- 距离度量算法
			  collapsed:: true
				- | 算法名称 | 核心作用 | 对异常值敏感 | 典型使用场景 |
				  | ---- | ---- | ---- |
				  | [[欧氏距离]] | 两个点各维度差值的平方和开平方 | 高 | 1. 低维、连续数值型数据（如身高、体重）2. [[KNN]] 、 [[K-Means]] 等基础聚类 / 分类算法3. 特征已做归一化 / 标准化、无明显异常值的场景 |
				  | [[曼哈顿距离]] | 两个点各维度差值的绝对值之和（城市街区距离） | 中 | 1. 高维稀疏数据、特征维度差异大的场景2. 网格状路径规划（如城市导航）3. 特征量纲一致、需要抗轻微异常值的场景 |
				  | [[切比雪夫距离]] | 两个点各维度差值的绝对值的最大值（也叫棋盘距离) | 极高（只看最大差） | 1. 物流配送、棋盘路径规划等场景2. 只需要关注「最大差异维度」的场景3. 特征维度重要性差异极大的场景 |
				  | [[闵可夫斯基距离]] | 欧氏、曼哈顿、切比雪夫距离的**通用公式**，通过参数 p 统一三种距离 | 随 p 值变化p=1 → 曼哈顿距离；p=2 → 欧氏距离；p→∞ → 切比雪夫距离； | 1. 需要灵活切换距离度量的场景2. 统一管理欧氏 / 曼哈顿 / 切比雪夫距离 |

- 企业级「业务问题→算法选型→工程实现」完整闭环实战指南
  collapsed:: true
	- 第一步：业务问题拆解与定义（最关键，决定后续所有方向）
	  collapsed:: true
		- 1. 核心动作：把业务需求转化为机器学习任务
			- 企业里 90% 的算法落地失败，都是因为**没把业务问题转成标准的 ML 任务**。
			  | 业务场景 | 拆解后的标准任务 | 对应学习方式 |
			  | ---- | ---- | ---- |
			  | 预测用户下个月的消费金额 | 回归任务（预测连续数值） | 有监督学习 |
			  | 判断用户是否会流失 | 分类任务（二分类：是 / 否） | 有监督学习 |
			  | 给用户做分群运营 | 聚类任务（无标签分组） | 无监督学习 |
			  | 智能推荐商品 | 排序 / 协同过滤（可归为分类 / 回归） | 有监督 / 强化学习 |
			  | 异常交易检测 | 分类任务（二分类：正常 / 异常） | 有监督 / 半监督学习 |
		- collapsed:: true
		  2. 关键决策点（直接影响算法选型）
			- 数据标签是否存在？
			  collapsed:: true
				- 有历史标签（如历史流失用户数据）→ 走有监督学习
				- 无标签（如全新业务无历史数据）→ 走无监督 / 半监督学习
			- 输出要求是什么？
			  collapsed:: true
				- 要具体数值（如消费金额）→ 回归任务
				- 要类别判断（如是否流失）→ 分类任务
			- 业务约束是什么？
			  collapsed:: true
				- 实时性要求（如推荐系统需要毫秒级响应）
				- 可解释性要求（如金融风控需要模型可解释）
				- 数据量大小（小样本 / 大数据）
				- 部署环境（端侧 / 云端）
		- collapsed:: true
		  3. 企业实战避坑
			- 不要上来就选复杂模型（如大模型、深度学习），先**用最简单的基线模型（如逻辑回归、线性回归）跑通**，再迭代优化
			- 必须和业务方对齐**评估指标**（如风控模型不能只看准确率，要看召回率、精确率），避免模型指标好看但业务没用
			  
			  ---
	- 第二步：算法选型（对应你梳理的 3 个「身份标签」）
		- 1. 选型逻辑：严格按「标签」层层筛选
			- ① 用「学习方式」标签定大方向
			  collapsed:: true
				- 有监督（带标签）→ 分类 / 回归任务
				- 无监督（无标签）→ 聚类 / 降维任务
				- 半监督（少量标签）→ 半监督分类 / 聚类
				- 强化学习（交互试错）→ 决策类任务（如智能调度、游戏 AI）
			- ② 用「任务适配」标签筛可用模型
			  collapsed:: true
				- 以**有监督 - 分类任务**为例，筛选出：逻辑回归、KNN、SVM、神经网络
				- 以**有监督 - 回归任务**为例，筛选出：线性回归、KNN、SVR、神经网络
			- ③ 用「核心特性」标签定最终选型
			  collapsed:: true
				- | 模型 | 核心特性 | 企业适用场景 | 不适用场景 |
				  | ---- | ---- | ---- |
				  | [[逻辑回归]] | 可解释性强、训练快、轻量 | 金融风控、广告点击率预估（需要可解释） | 非线性复杂数据、高维稀疏数据 |
				  | [[线性回归]] | 可解释性强、实现简单 | 销量预测、房价预测（线性关系明显） | 非线性复杂数据 |
				  | [[KNN]] | 无训练、实现简单 | 小样本、低维数据、冷启动场景 | 大数据、高维数据、实时性要求高 |
				  | [[SVM/SVR]] | 小样本高维数据表现好 | 文本分类、图像分类（小样本） | 大数据量、实时性要求高 |
				  | [[神经网络]] / 深度学习 | 拟合能力强、端到端 | 计算机视觉、NLP、复杂非线性数据 | 小样本、需要可解释性、资源有限 |
		- 2. 企业选型黄金原则
			- 基线优先：先跑通逻辑回归 / 线性回归，再上复杂模型
			- 业务适配：金融风控优先可解释性（逻辑回归），推荐系统优先效果（深度学习）
			- 资源匹配：小团队 / 资源有限，优先轻量模型；大厂 / 资源充足，可上深度学习
			- 迭代优化：不是一次选对，而是先上线，再用 A/B 测试迭代优化
	- 第三步：工程实现（从模型到线上服务的全流程）
		- 1. 标准工程流程（企业级 ML 流水线）
			- ① 数据预处理（对应你学的「特征预处理」）
				- 数据清洗：处理缺失值、异常值、重复值
				- 数据集划分：训练集、验证集、测试集（7:2:1 或 8:1:1）
				- 特征工程：特征选择、特征构建、归一化 / 标准化（距离类模型必须做）
				- 交叉验证：用 CV 评估模型稳定性，避免过拟合
			- ② 模型训练与调参（对应你学的「工具层」）
			  collapsed:: true
				- 基线模型训练：用默认参数跑通，得到 baseline 指标
				- 超参数调优：用网格搜索 / 随机搜索 + CV，优化超参数
				- 模型评估：用对应指标（分类：准确率、召回率、F1；回归：MAE、MSE、RMSE）
				- 过拟合 / 欠拟合处理：正则化、增加数据、调整模型复杂度
			- ③ 模型部署与服务化
			  collapsed:: true
				- 模型导出：保存训练好的模型（如 pickle、ONNX 格式）
				- 服务封装：用 FastAPI/Flask 封装成 HTTP 接口，供业务系统调用
				- 部署环境：云端（AWS、阿里云）、容器化（Docker+K8s）、端侧（移动端 / 嵌入式）
				- 监控运维：监控模型性能、数据漂移、服务可用性，定期迭代模型
		- 2. 企业工程实现核心要点
			- 可复用性：搭建标准化 ML 流水线，避免重复造轮子（如用 Airflow 调度、MLflow 管理模型）
			- 可监控性：必须监控数据漂移（数据分布变化导致模型失效），这是企业模型上线后最大的坑
			- 可扩展性：支持模型迭代、A/B 测试、灰度发布，不影响业务
			- 合规性：金融、医疗等行业，必须满足数据合规、模型可解释性要求
