- **一句话定义**
	- 上下文工程是系统性管理大模型输入上下文的工程方法，包括 Prompt、RAG 证据、状态、记忆、工具结果、结构化输出和安全约束。

- **解决的问题**
	- 大模型上下文窗口有限，不能把所有信息直接塞进去。
	- 输入内容来源复杂，可能包含用户问题、历史对话、检索证据、工具结果和系统约束。
	- 上下文顺序、噪声、冲突和缺失都会影响模型输出。

- **和 Prompt Engineering 的区别**
	- Prompt Engineering 更关注“如何写指令”。
	- 上下文工程更关注“系统应该给模型什么信息、以什么结构给、给多少、何时更新、如何过滤”。
	- 在生产系统中，上下文工程通常包含 RAG、状态管理、记忆、工具调用结果、权限和输出约束。

- **核心组成**
	- 指令：系统规则、角色、任务边界。
	- 证据：RAG 检索到的 chunk、引用、metadata。
	- 状态：历史对话、当前任务阶段、用户偏好。
	- 工具结果：API、数据库、搜索、代码执行结果。
	- 约束：安全、权限、格式、拒答条件。

- **项目中怎么用**
	- RAG 系统中，上下文工程决定哪些证据进入 Prompt、顺序如何排、冲突如何处理。
	- Agent 系统中，上下文工程决定任务状态、工具结果和下一步计划如何进入模型。

- **面试表达**
	- 上下文工程和 Prompt Engineering 有什么区别？ #card
		- Prompt Engineering 关注指令怎么写；上下文工程关注整个输入上下文怎么组织，包括指令、检索证据、历史状态、工具结果、记忆和约束。生产系统更需要上下文工程。

- **常见误区**
	- 误区：上下文越多越好。
		- 上下文过多会引入噪声、冲突和 Lost in the Middle。
	- 误区：只要改 Prompt 就能提升真实性。
		- 如果上下文证据不正确或顺序混乱，生成约束很难稳定生效。
