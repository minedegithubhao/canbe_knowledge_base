- **一句话定义**
	- RAG 是一种“先检索、再生成”的大模型应用架构：系统先从外部知识库召回和问题相关的证据，再把证据放进 Prompt，让 LLM 基于这些材料生成答案。

- **解决的问题**
	- 模型知识过时：外部知识库可以更新，不必重新训练大模型。
	- 私有知识接入：企业文档、FAQ、手册、工单、内部制度可以通过检索进入上下文。
	- 幻觉风险：模型回答时有证据约束，但 RAG 只能降低幻觉，不能完全消除幻觉。
	- 可追溯性：答案可以附带来源片段、章节、文档 ID 或 FAQ ID，方便人工审核。

- **标准流程**
	- 离线入库
		- 收集原始文档。
		- 用 [[Concept/文档清洗]] 去掉噪声并保留业务语义。
		- 用 [[Concept/文档切分]] 切成适合检索和生成的 chunk。
		- 用 [[Concept/文本嵌入]] 生成 embedding，同时保留标题、章节、来源、时间等 metadata。
		- 写入 [[Concept/向量库]]、搜索引擎或混合检索系统。
	- 在线问答
		- 用户输入问题。
		- 必要时用 [[Concept/Query Rewriting]] 做改写、补全指代、术语标准化。
		- 用向量检索、关键词检索或 [[Concept/混合检索]] 召回候选 chunk。
		- 用 [[Concept/重排序]] 精排，过滤低相关内容。
		- 必要时通过 [[Concept/检索后过滤]] 触发拒答、追问或降级处理。
		- 构建 Prompt，要求模型基于证据回答。
		- 输出答案、引用、拒答或追问。

- **关键取舍**
	- 召回率和准确率通常需要权衡：Top K 提高可能带来更多相关信息，也可能引入噪声。
	- chunk 太小可能语义不完整，chunk 太大可能信噪比下降。
	- 长尾查询会暴露热门 FAQ 覆盖不到的低频实体、复杂条件和口语化表达问题。
	- 向量检索适合语义相近的问题，关键词检索适合编号、实体名、专业术语和精确匹配。
	- 关键词检索依赖 [[Concept/倒排索引]] 和分词策略，编号、符号、字段设计都会影响效果。
	- Rerank 可以提高上下文纯度，但阈值过高可能过滤掉弱相关但必要的信息。
	- Prompt 不是替代检索的手段；如果证据没召回，生成阶段很难稳定补救。

- **面试表达**
	- 什么是 RAG？ #card
		- RAG 是检索增强生成。它先从外部知识库召回和问题相关的证据，再把证据放进 Prompt，让大模型基于这些材料回答，从而缓解知识过时、私有知识接入和事实幻觉问题。
	- RAG 的完整流程是什么？ #card
		- 离线阶段做文档清洗、切分、向量化和入库；在线阶段对 query 做处理，通过向量检索、关键词检索或混合检索召回候选，再用 Rerank 精排，最后把证据放进 Prompt 让 LLM 生成答案。
	- RAG 使用后仍然幻觉，怎么排查？ #card
		- 先看召回结果是否包含正确证据。如果没有，排查清洗、切分、Query Rewriting、检索策略、Top K 和 Rerank。如果已经召回正确证据，再排查上下文排序、Prompt 约束、引用策略和模型是否严格基于资料回答。

- **常见误区**
	- 误区：接入向量库就等于做好 RAG。
		- 实际上 RAG 的效果依赖清洗、切分、召回、融合、重排序、Prompt、评估和数据治理。
	- 误区：RAG 可以完全消除幻觉。
		- RAG 只能提供外部证据并约束生成，召回错误、证据不足或 Prompt 约束弱时仍会幻觉。
	- 误区：chunk 越大越好。
		- chunk 大会增加语义完整性，也可能降低信噪比；chunk 小更精确，也可能丢失上下文。
