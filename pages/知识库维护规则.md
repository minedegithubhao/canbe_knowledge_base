- ```text
  基于 AGENTS.md 和 logseq-knowledge-refactor 规则，整理这个新知识点：xxx。
  请先查重，再判断应创建或更新 Source / Concept / Project / Question / Review / Mistake。
  保留来源链路，更新相关 MOC 和 Review，最后做 Logseq 格式与发布风险检查。
  ```
- **知识库目标**
	- 这个知识库采用 Karpathy LLM-Wiki 思路维护。
	- 原始资料保留在 `Source` 页面中，稳定知识沉淀到 `Concept` 页面中，主题导航由 `MOC` 页面负责。
	- 目标不是堆积笔记，而是形成一个便于阅读、复习、项目复用、LLM 持续维护的知识系统。
	- `AGENTS.md` 是 LLM 执行时的唯一标准，本页面是给人在 Logseq 中阅读的摘要。
- **核心流程**
	- 新资料进入知识库时，遵循：
		- `Source -> Concept -> MOC -> Review`
	- `Source` 记录资料说了什么。
	- `Concept` 记录我综合理解后的稳定知识。
	- `MOC` 提供主题入口和学习路径。
	- `Review` 服务刷题和复习。
- **页面类型**
	- `home`：知识库首页。
	- `moc`：主题地图，对应 [[MOC/RAG]]、[[MOC/大模型]]。
	- `concept`：稳定概念，对应 [[Concept/RAG]]、[[Concept/混合检索]]。
	- `source`：原始资料，对应 `Source` 页面。
	- `project`：项目沉淀，对应 `Project` 页面。
	- `question`：复杂面试题或项目追问，对应 `Question` 页面。
	- `review`：复习页或刷题题单，对应 `Review` 页面。
	- `mistake`：错题或易错点，对应 `Mistake` 页面。
- **命名规则**
	- 页面 namespace 使用英文标准分类；页面正文、解释和 alias 使用中文友好表达。
	- 页面名使用人最自然会点击的名字。
	- alias 用来补齐中文名、英文名、全称、简称、常见别名和旧页面名。
	- 统一使用英文 namespace：
		- [[MOC/大模型]]
		- [[Concept/RAG]]
		- [[Source/Karpathy LLM Wiki]]
		- [[Project/图文知识库]]
		- [[Question/RAG 为什么还会幻觉]]
		- [[Review/RAG面试]]
		- [[Mistake/RAG易错点]]
	- [[首页]] 是唯一命名例外，对应 home 类型。
	- 不创建 [[Home/首页]] 或其他 Home namespace 页面。
	- `MOC` 页面中文主题名为主，例如 [[MOC/大模型]]、[[MOC/面试复习]]。
	- `Concept` 页面使用术语自然名称。
		- RAG、LLM、Transformer、Prompt、Token、Embedding 等常用英文术语保留英文。
		- 混合检索、文档切分、向量检索等中文更自然的概念使用中文。
	- `Source` 页面使用资料标题或可识别的短标题。
	- `Source` 命名优先使用作者或机构加短标题，必要时加年份，避免同名资料冲突。
	- `Project` 页面使用项目中文名或实际项目名。
	- `Question` 页面使用中文问题句。
	- `Review` 页面使用中文题单名。
	- 推荐：
		- [[首页]]
		- [[MOC/RAG]]
		- [[MOC/大模型]]
		- [[Concept/RAG]]
		- [[Concept/混合检索]]
		- [[Source/Karpathy LLM Wiki]]
		- [[Project/图文知识库]]
		- [[Question/RAG 为什么还会幻觉]]
		- [[Review/RAG面试]]
	- 避免创建过深 namespace，例如：
		- `[[大模型/RAG/文本聚类/算法/无监督聚类算法]]`
	- 不新建中文分类 namespace 页面，例如 `[[地图/RAG]]`、`[[概念/RAG]]`。
	- 页面名尽量短，关系通过 `domain`、`related` 和双链表达。
	- 不为同一概念同时创建中文页和英文页；选择一个主页面，其他名称放入 alias。
	- 旧页面迁移到新页面后，将旧页面名加入新页面 alias，方便搜索和去重。
	- alias 是为了搜索和去重，不是词典，不塞大量低频别名。
	- 普通属性保持单值；alias 是允许多值的例外；related、sources、questions 这类多关系放正文区块。
	- alias 优先使用 Logseq 页面名和自然语言别名，例如 `大模型/RAG`，不要优先使用文件名形态 `大模型___RAG`。
	- 不为套用模板而编造内容；未知、不确定、未验证的内容留空，或写入 **待验证**。
	- 创建新页面前必须查重：
		- 搜索目标页面名。
		- 搜索中文名、英文名、缩写和常见别名。
		- 搜索现有 alias。
		- 搜索旧 namespace，例如 `大模型___RAG` 或 `大模型/RAG`。
		- 如果已有同义页面，优先更新、迁移或合并，不新建重复页。
	- 旧页面迁移时使用 **迁移说明** block，不依赖 redirect 属性实现自动跳转。
- **格式规则**
	- 页面正文使用 Logseq block。
	- 区块标题统一使用：
		- `- **区块名**`
	- 不使用：
		- `- ## 区块名`
	- 页面内部尽量不使用传统 Markdown 标题 `#`、`##`、`###`。
	- 一个属性只写一个值。
	- alias 是多值例外。
	- 面试卡片继续使用 `#card`。
- **状态和日期规则**
	- 日期统一使用 `YYYY-MM-DD`。
	- created 表示页面创建日期。
	- updated 表示页面最后整理日期。
	- Source 页面可额外记录 published 和 accessed。
	- Source 状态：
		- raw
		- summarized
		- compiled
		- archived
	- Source 状态流转：
		- raw：只有原始链接、附件、摘录或截图。
		- summarized：已经完成 Source 页面的核心观点总结。
		- compiled：已经更新相关 Concept、MOC、Question 或 Project，并在 **已更新页面** 中记录。
		- archived：资料不再主动维护，但保留引用。
	- Concept 状态：
		- seed
		- growing
		- stable
		- review
		- archived
	- Question 状态：
		- seed
		- practicing
		- weak
		- mastered
		- archived
	- Review 状态：
		- growing
		- active
		- paused
		- archived
	- Project 状态：
		- planning
		- active
		- done
		- archived
	- Home 状态：
		- active
		- archived
	- Mistake 状态：
		- weak
		- practicing
		- mastered
		- archived
- **隐私和发布规则**
	- 默认页面可以发布。
	- `journals/` 下的日报、日记默认不发布。
	- 页面包含 private 标记时不发布。
	- 私有内容统一优先使用 tags private。
	- LLM 不主动移除隐私标记。
	- LLM 不应把包含隐私标记的内容复制到公开页面，除非明确要求并确认可公开。
	- 公开页面不要链接 private 页面，因为页面名本身也可能泄露信息。
	- 如果公开页面必须引用 private 页面，应将公开页面也标记为 private，或改成不暴露敏感名称的普通描述。
	- 发布到 GitHub Pages 前会运行 `scripts/prepare-publish-graph.ps1` 生成 `.publish-graph`。
	- 发布脚本默认不包含 `journals/` 和 `whiteboards/`。
	- 发布脚本只复制必要的 Logseq 配置，不复制 `logseq/bak`、`logseq/.recycle` 等历史备份或回收站内容。
	- 发布脚本会删除 private Markdown 页面，并移除未被公开页面引用的 `assets/` 文件。
	- 不要在公开页面引用私有附件；页面从公开改成 private 后，应检查其附件是否仍被其他公开页面引用。
- **图片和图示规则**
	- 图片、截图、PDF、附件统一放在 `assets/`，不要放进 `pages/`。
	- 普通来源截图、文章配图、PDF 附件放在 `assets/sources/`。
	- 项目相关图片放在 `assets/projects/`。
	- 导出的架构图、流程图、概念图放在 `assets/diagrams/`。
	- Logseq 白板文件保留在 `whiteboards/`。
	- LLM 默认不创建、不修改 `whiteboards/` 下的 `.edn` 文件。
	- LLM 可以阅读白板作为参考，但不主动维护白板。
	- 只有明确要求编辑白板时，才修改 `whiteboards/`。
	- Mermaid 是文本图，直接写在页面正文中，不需要放入 `assets/`。
	- 只有当图能明显降低理解成本时才画图。
	- `Concept` 页面适合放核心机制图或结构图。
	- `Project` 页面适合放架构图、流程图、模块关系图。
	- `MOC` 页面适合放学习路径图。
	- `Question` 页面一般不画图，除非是复杂项目追问。
	- `Source` 页面一般只保留资料原图或截图，不主动重画。
	- 每张图前后都要有文字说明，图不能替代概念解释。
- **证据和冲突处理**
	- `Concept` 页面写综合结论，但重要结论必须能追溯到 `Source`。
	- 页面底部 **来源** 保留来源列表。
	- 对关键判断、项目经验、易过期信息，应在相关 block 中直接链接来源。
	- 不要用新资料直接覆盖旧结论。
	- 如果不同资料说法冲突，新增 **不同说法** 或 **争议点** 区块说明差异。
	- 当前结论需要明确是综合判断，并链接支撑来源。
	- AI 模型、框架、库版本、价格、政策、法规等容易过期的信息，需要记录发布日期或访问日期。
	- 不确定、不完整、未验证的信息不要写成确定结论，放入 **待验证** 区块。
	- 涉及最新模型、价格、API、法规、政策、版本变化的信息，必须链接来源并记录访问日期。
- **Concept 页面结构**
	- 推荐结构：
		- alias ：页面别名
		- type ：concept
		- domain ：所属主题
		- status ：seed
		- level ：掌握程度
		- created ：创建日期
		- updated ：更新日期
	- 正文区块：
		- **一句话定义**
		- **解决的问题**
		- **核心机制**
		- **典型流程**
		- **项目中怎么用**
		- **面试表达**
		- **常见误区**
		- **相关概念**
		- **来源**
	- 不为填满模板而编造内容；未知内容留空或放入 **待验证**。
- **Home 页面结构**
	- `Home` 是知识库总入口，不写具体知识解释。
	- [[首页]] 是唯一 Home 页面。
	- 推荐区块：
		- **主要入口**
		- **当前重点**
		- **待整理**
		- **最近更新**
		- **复习入口**
		- **维护规则**
	- `Home` 应链接重要 `MOC`、`Project`、`Review` 和维护规则。
	- `Home` 保持短，不要变成大杂烩。
- **Source 页面结构**
	- 推荐结构：
		- type ：source
		- source-type ：article / paper / video / course / book / project / image
		- author ：作者
		- url ：资料链接
		- published ：资料发布日期
		- accessed ：访问日期
		- created ：创建日期
		- updated ：整理日期
		- status ：raw
	- 正文区块：
		- **原始信息**
		- **核心观点**
		- **可沉淀的概念**
		- **已更新页面**
		- **待追问问题**
	- `Source` 页面只记录单个资料，不写跨资料综合结论。
	- 资料标题过长时，使用作者或机构加短标题命名，例如 [[Source/Karpathy - LLM Wiki]]。
- **MOC 页面结构**
	- `MOC` 是导航页，只放入口、路径、索引和少量说明。
	- 推荐区块：
		- **学习路径**
		- **核心概念**
		- **项目实战**
		- **面试重点**
		- **最近更新**
		- **尚未整理**
	- 如果一个 `MOC` 页面中超过约 30% 是解释性内容，应迁移到对应 `Concept` 页面。
- **Project 页面结构**
	- `Project` 页面沉淀项目上下文、方案、取舍和复盘。
	- 推荐区块：
		- **项目目标**
		- **背景资料**
		- **系统架构**
		- **核心流程**
		- **关键模块**
		- **问题记录**
		- **经验沉淀**
		- **相关概念**
		- **来源**
	- 项目中产生的稳定知识应回流到相关 `Concept`。
	- 项目中常被问到的问题应沉淀为 `Question`。
- **Mistake 页面结构**
	- `Mistake` 页面记录真实错题和易错点，不记录普通知识点。
	- 推荐区块：
		- **错误题目**
		- **我的错误答案**
		- **正确答案**
		- **错因分析**
		- **关联概念**
		- **复习计划**
		- **来源**
	- 错因必须链接回相关 `Concept` 或 `Question`。
	- `Mistake` 是个人错误记录；通用题目本身应放在 `Question`。
- **面试题规则**
	- 简单定义题放在 `Concept` 页的 **面试表达** 下，并标记 `#card`。
	- 复杂题、项目题、易错题、追问题单独创建 `Question` 页面。
	- `Question` 记录可复用的面试题、项目追问或练习题。
	- 真实答错、做错或反复混淆的内容放入 `Mistake`，并链接对应 `Question` 或 `Concept`。
	- 刷题入口放在 `Review` 页面。
	- 面试复习总入口放在 [[MOC/面试复习]]。
	- 推荐链路：
		- [[MOC/面试复习]]
		- [[Review/RAG面试]]
		- [[Question/RAG 为什么还会幻觉]]
		- [[Concept/RAG]]
	- **日常维护流程**
		- 新增外部资料时，先创建或更新 `Source` 页面，记录来源、访问日期、核心观点和已更新页面。
		- 稳定知识沉淀到 `Concept`，并在 **来源** 中保留来源链路。
		- 项目实践中的背景、流程、问题、指标和取舍沉淀到 `Project`，不要塞进 `Concept` 正文。
		- 面试复习新增内容时，先判断是短卡片还是复杂题。
		- 短定义题放入对应 `Review`，或放入 `Concept` 的 **面试表达**。
		- 复杂追问、项目题、容易混淆的题，创建或更新 `Question` 页面。
		- 真实答错、反复混淆或复盘过的问题，创建或更新 `Mistake` 页面，并链接对应 `Question` 和 `Concept`。
		- 每次维护结束，至少更新相关 `MOC`、`Review` 和 updated 日期。
		- 零散想法、临时摘录、待办可以先放当天 journal，但不要把 journal 当成长期知识页。
		- 每周或每个主题批次结束后，检查 [[首页]]、[[MOC/面试复习]] 和相关主题 MOC 是否仍然好扫。
		- 发布到 GitHub Pages 前，检查 private 标记、公开页面链接、本地路径、内网地址、凭据、未引用附件和 pages 下二进制文件。
		- 能小步维护时优先小步维护；只有同一主题出现大量重复、旧 namespace 或面试链路断裂时，才启动主题重构。
- **旧页面重构时**
	- 先判断旧页面属于哪种类型。
	- 内容完整的页面升级为新模板。
	- 过细或重复的页面合并到更合适的 `Concept`。
	- 空页面或低价值页面将 status 设为 archived。
	- 不直接删除旧页面，除非确认不再需要。
	- 迁移后保留跳转说明，避免旧双链失效。
	- 每次优先只重构一个主题，例如只处理 RAG 或只处理 Transformer。
	- 涉及超过 10 个页面的重命名、合并、迁移、归档，必须先列迁移计划并等待确认。
	- 迁移计划应包含旧页面、新页面、处理方式、是否保留迁移说明。
	- 不批量删除页面。
	- 不修改 `journals/` 中的历史记录，除非明确要求。
- **修改后自检**
	- 检查新增或修改页面是否有正确的 type、status、updated。
	- 检查是否误用了 `- ## 区块名`。
	- 检查是否在 Logseq 页面正文中误写属性示例。
	- 检查新建页面前是否已经查重，避免重复概念页。
	- 检查关键结论是否有 `Source` 支撑。
	- 检查是否误改 `whiteboards/` 或 `journals/`。
	- 检查是否移除了 private 标记。
	- 自检优先针对本次新增或修改的页面。
	- 全库扫描可能发现历史遗留问题，不把无关旧页面作为当前任务阻塞项，也不顺手批量修复无关旧页面。
- **质量标准**
	- 打开页面后，能快速知道这个知识点是什么。
	- 页面能说明它解决什么问题、怎么工作、项目中怎么用。
	- 重要结论能追溯到 `Source`。
	- 面试内容能直接用于复习。
	- 页面与相关概念互相链接，不孤立存在。
	- 格式稳定，LLM 下次能继续维护。
- **Agent 执行规则**
	- LLM 维护知识库时，优先阅读根目录的 `AGENTS.md`。
	- 本页面是给人在 Logseq 中阅读的维护手册。
