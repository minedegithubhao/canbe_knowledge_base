- **项目目标**
	- 建立一个能管理文本、图片、图文页面和相关知识链接的 RAG 知识库项目入口。

- **背景资料**
	- 这是 RAG 主线中的多模态和图文方向项目页。
	- 白板由人维护，LLM 默认只维护 Markdown 页面和 Mermaid 图，不直接修改 `whiteboards/*.edn`。

- **系统架构**
	- 文本层：普通文档、标题、正文、metadata。
	- 图片层：截图、流程图、架构图、OCR 文本、caption。
	- 检索层：文本检索、图片检索、图文对齐检索。
	- 融合层：多路召回、结果融合、重排序。
	- 生成层：基于文本和图片证据回答。

- **核心流程**
	- 图文文档解析。
	- 图片 OCR 和 caption 生成。
	- 文本向量和图片向量入库。
	- 多路召回文本和图片证据。
	- 融合和重排序。
	- 回答时保留来源和引用。

- **问题记录**
	- 图中的关键信息可能不会出现在正文里。
	- OCR 和 caption 可能丢失流程关系或表格结构。
	- 文本和图片证据需要融合排序，不能只拼接。
	- 私有图片和附件发布前需要检查隐私规则。

- **指标和验收**
	- 文本证据命中率：问题所需文字证据是否能被召回。
	- 图片证据命中率：图中关键信息是否能通过 OCR、caption 或图片检索被召回。
	- 图文融合质量：文本证据和图片证据是否能一起支撑答案，而不是互相冲突。
	- 来源可追溯性：回答是否能回到原文、图片、页面或附件。
	- 隐私发布检查：公开页面不引用 private 图片或不可发布附件。

- **经验沉淀**
	- 图文 RAG 的第一步不是追求全模态，而是把 OCR、caption、metadata 和来源管理做好。
	- 白板适合人手动维护，LLM 默认不直接编辑 `whiteboards/*.edn`。
	- Mermaid 适合 LLM 维护的结构图，白板适合人工探索。

- **面试表达**
	- 图文知识库 RAG 项目怎么讲？ #card
		- 重点讲图文证据融合。图中的信息不一定出现在正文里，所以要处理 OCR、caption、图片 metadata、页面上下文、文本检索、图片检索和融合重排，回答时保留来源。
	- 为什么不能只把图片 OCR 后当文本处理？ #card
		- OCR 只能提取文字，流程关系、空间结构、表格布局和图片语义可能会丢失，还需要 caption、metadata、页面上下文和必要的图像检索补充。
