- **一页总览**
	- RAG 是架构，不是一个模型。
	- 离线入库：清洗、切分、embedding、索引、metadata。
	- 在线问答：query 处理、召回、融合、Rerank、Prompt、生成、引用或拒答。
	- 优化顺序：先看评估日志，再判断问题在数据、切分、召回、重排、Prompt 还是模型。

- **高频必会**
	- 什么是 RAG？ #card
		- RAG 是检索增强生成。它先从外部知识库召回和问题相关的证据，再把证据放进 Prompt，让大模型基于这些材料回答，从而缓解知识过时、私有知识接入和事实幻觉问题。
	- RAG 解决什么问题？ #card
		- 主要解决三类问题：模型知识过时、无法直接使用企业私有数据、事实性回答容易幻觉。同时还能提高答案可追溯性，因为答案可以附带引用来源。
	- RAG 的完整流程是什么？ #card
		- 离线阶段做文档清洗、切分、embedding、索引和 metadata 入库；在线阶段做 query 处理，通过向量检索、关键词检索或混合检索召回候选，再做融合和 Rerank，最后把证据放进 Prompt 让 LLM 生成答案。
	- RAG 为什么不能完全解决幻觉？ #card
		- 因为 RAG 只是给模型提供外部证据和约束，最终仍由模型生成文本。如果召回错误、证据不足、上下文噪声大、Prompt 约束弱，模型仍可能编造。
	- RAG 幻觉应该怎么排查？ #card
		- 先看召回结果是否包含正确证据。没有证据就排查清洗、切分、Query Rewriting、检索策略、Top K 和 Rerank；有证据再排查上下文顺序、Prompt 约束、引用要求和模型遵循能力。

- **检索链路**
	- 文档清洗为什么重要？ #card
		- 清洗是为了去掉页眉页脚、目录、重复内容、乱码、无意义换行等噪声，同时保留标题、表格、FAQ、编号、条件和业务实体，让后续切分、embedding、BM25、Rerank 和生成更稳定。
	- 文档切分为什么重要？ #card
		- chunk 是 RAG 检索和生成的基本单位。切得太小会语义不完整，切得太大又会混入噪声，影响召回准确率、Rerank 和最终答案质量。
	- 你会怎么设计文档切分策略？ #card
		- 先看用户问题和答案粒度，再利用标题、段落、列表、表格、FAQ 等结构做语义边界；超过 token 限制再二次切分，并保留标题、章节、来源、版本等 metadata。
	- chunk size 越大越好吗？ #card
		- 不一定。适度增大能提高语义完整性，过大则会稀释关键词、降低信噪比，影响检索准确性和生成可靠性。要结合评估集看召回率、准确率和真实性。
	- 为什么 RAG 要关注长尾查询？ #card
		- 因为线上真实问题往往低频、多样、具体。只优化热门问题会让评估虚高，但系统在罕见实体、复杂约束和口语化表达上仍然容易漏召回或幻觉。
	- overlap 的作用是什么？ #card
		- overlap 用来缓解语义被切断的问题，让相邻 chunk 共享部分上下文；但 overlap 过大会带来重复内容和检索噪声。
	- embedding 在 RAG 里解决什么问题？ #card
		- embedding 把 query 和 chunk 转成向量，让系统可以按语义相似度召回文档片段。但它会损失细节和结构，所以常需要关键词检索、metadata 和 Rerank 补充。
	- 向量检索和关键词检索分别适合什么场景？ #card
		- 向量检索适合同义表达、口语化和语义相似问题；关键词检索适合编号、产品名、术语、实体名、函数名和精确匹配。
	- 为什么要做混合检索？ #card
		- 因为向量检索和关键词检索互补。向量检索能抓语义相似，关键词检索能抓精确词、编号和专有名词，混合检索能提高召回稳定性。
	- 倒排索引在 RAG 里有什么用？ #card
		- 它支撑关键词检索和 BM25，弥补向量检索对精确词、编号、实体名不稳定的问题，通常和向量检索一起组成混合检索。
	- RRF 是什么，和 Rerank 有什么区别？ #card
		- RRF 是融合算法，按多个召回列表的排名合并结果，不理解 query 和 chunk 的语义；Rerank 是精排模型，会重新判断 query 和 chunk 的相关性。RRF 常用于多路召回融合，Rerank 用于最终证据筛选。
	- 为什么 RAG 需要 Rerank？ #card
		- 初召回通常追求覆盖率，会带来弱相关和噪声证据。Rerank 用更强的相关性模型精排候选，提高上下文纯度，但它不能弥补初召回没有正确候选的问题。
	- BGE Reranker 在 RAG 中怎么用？ #card
		- 它放在初召回之后、Prompt 构建之前，对 query 和候选 chunk 成对打分，再按分数排序、过滤低分证据，最终保留 Top K 进入上下文。
	- 自查询检索解决什么问题？ #card
		- 它从用户问题中提取章节、类别、时间、版本等 metadata 条件，用过滤缩小检索范围，适合手册、FAQ、制度等结构化文档。
	- Query Rewriting 解决什么问题？ #card
		- 它把口语化、省略、指代不清、术语不统一的问题改写成更适合检索的标准问题，减少检索偏移。
	- Query Rewriting 有什么风险？ #card
		- 风险是改写漂移。模型可能加入用户没说过的信息或过度扩展问题，导致检索偏移。要约束保留原意、不新增事实，并用评估集验证。
	- Rerank 放在哪一步，解决什么问题？ #card
		- Rerank 放在初召回和 Prompt 构建之间，用更强的相关性模型对候选 chunk 重新打分，保留最相关证据，降低上下文噪声。
	- 检索后过滤如何降低幻觉？ #card
		- 它在生成前检查证据是否足够相关。如果没有可信证据，就拒答、追问或转人工，避免模型在没有上下文支撑时硬答。

- **评估优化**
	- 如何评估 RAG 效果？ #card
		- 可以拆成检索和生成两段。检索看召回率、准确率和命中证据；生成看答案是否忠实证据、是否回答问题、是否覆盖关键点、是否简洁。
	- 召回率低怎么排查？ #card
		- 先确认知识库里有没有正确答案，再看清洗和切分是否保留证据，然后检查 query、检索策略、metadata 过滤、Top K 和 Rerank 阈值，定位正确证据在哪一步丢失。
	- 准确率低怎么排查？ #card
		- 如果答案所需证据被召回了但混入很多无关内容，重点看 chunk 粒度、Top K、多路融合、metadata 过滤和 Rerank，而不是直接改 Prompt。
	- 真实性低怎么排查？ #card
		- 先看上下文是否有正确且干净的证据。如果没有，优化检索；如果有，再加强 Prompt 约束、引用格式和拒答策略。
	- Top K 越大越好吗？ #card
		- 不一定。Top K 变大可能提高召回率，也可能引入更多噪声，降低答案真实性和简洁度，需要结合评估集调参。
	- Rerank 阈值越高越好吗？ #card
		- 不一定。阈值高能提高上下文纯度，但可能过滤掉弱相关但必要的证据。阈值要看评估日志和错误样本。
	- RAGAS 和 PAGAS 怎么讲？ #card
		- RAGAS 更偏技术评估，帮助定位检索和生成问题；PAGAS 更偏业务验收，关注答案是否有证据、是否相关、是否覆盖关键属性、是否简洁可用。
	- 相关性和真实性有什么区别？ #card
		- 相关性看答案是否回应问题，真实性看答案是否基于事实和证据。答案可能真实但没回答问题，也可能相关但事实错误。

- **项目追问**
	- 最小 RAG 系统需要哪些模块？ #card
		- 至少需要文档加载、文档切分、embedding、向量库、检索、Prompt 构建、LLM 生成和评估。生产系统还会加入混合检索、Rerank、Query Rewriting、metadata 和权限控制。

- **场景追问**
	- 用户说答案不对，但检索日志里有正确 chunk，怎么办？ #card
		- 优先检查上下文排序、chunk 是否太长、Prompt 是否要求基于证据回答、答案格式是否约束清楚，以及模型是否被无关上下文干扰。
	- 用户问的问题知识库里没有，RAG 应该怎么处理？ #card
		- 应该拒答或提示知识库未覆盖，而不是让模型凭常识回答。客服、制度和合规场景尤其要明确拒答策略。
	- 召回内容互相冲突怎么办？ #card
		- 先看 metadata，比如版本、时间、适用范围和来源优先级；Prompt 里也要要求优先使用最新或权威来源，并在必要时说明冲突。
	- 为什么不能只靠 Prompt 解决 RAG 问题？ #card
		- Prompt 只能约束生成阶段。如果正确证据没有召回，或者上下文里噪声过多，Prompt 无法稳定弥补检索链路的问题。

- **易错题**
	- [[Mistake/RRF 和 Rerank 混淆]]
	- 看到生成答案错了，为什么不能直接改 Prompt？ #card
		- 因为问题可能出在检索阶段。没有正确证据时，Prompt 再强也不稳定；应该先看召回日志，再判断是检索问题还是生成约束问题。
	- 摘要向量一定能提高召回吗？ #card
		- 不一定。摘要能降噪和强化语义特征，但如果问题需要精确编号、章节、术语或 metadata 过滤，摘要向量不一定能解决。
	- BM25 一定能解决关键词问题吗？ #card
		- 不一定。分词、编号格式、符号、大小写、同义词和字段设计都会影响 BM25，需要结合 analyzer、字段设计和 metadata 过滤。
	- 有了 Rerank，是不是就不用改切分？ #card
		- 不是。如果 chunk 被切断、缺少关键特征或混入过多噪声，Reranker 也可能给低分或无法正确判断。
	- 向量库是不是 RAG 的全部？ #card
		- 不是。向量库只是召回基础设施，RAG 还需要清洗、切分、索引、混合检索、Rerank、Prompt、评估和数据治理。
