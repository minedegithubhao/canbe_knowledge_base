- **一句话定义**
	- Query Rewriting 是把用户原始问题改写成更适合检索的标准查询，以提升 RAG 召回准确性。

- **解决的问题**
	- 用户问题口语化、啰嗦、带情绪或缺少主语。
	- 多轮对话里存在“它”“这个”“刚才那个”等指代。
	- 用户表达和文档术语不一致。
	- 问题过长、包含多个意图或混入无关信息。

- **典型流程**
	- 输入原始 query 和必要的会话历史。
	- 使用规则、Prompt 或小模型抽取真实意图。
	- 补全指代和省略信息。
	- 标准化业务术语、实体名和时间范围。
	- 输出一个或多个用于检索的 query。

- **好改写的标准**
	- 保留用户原始意图。
	- 不新增用户没有表达的事实。
	- 补全必要上下文，但不替用户做答案推断。
	- 输出简洁、可检索、术语稳定的问题。
	- 必要时保留原始 query 和改写 query 同时检索。

- **项目中怎么用**
	- 在 [[Project/小说RAG实战]] 中，把口语化问题改成包含角色名、时期、章节线索和专有名词的检索 query。
	- 在客服场景中，把“这个怎么退啊”改写为包含商品、订单、退款、售后条件的标准问题。
	- 对强模型改写要加约束，避免把用户问题文学化、推理化或引入知识库没有的事实。

- **面试表达**
	- Query Rewriting 解决什么问题？ #card
		- 它解决用户问题不适合直接检索的问题，例如口语化、省略、指代、术语不统一和多意图混杂。改写后 query 更接近知识库表达，召回更稳定。
	- Query Rewriting 有什么风险？ #card
		- 风险是改写模型可能加入用户没有说过的信息，导致检索偏移。因此改写规则要要求保留原意、不编造事实，必要时原始 query 和改写 query 同时检索。

- **常见误区**
	- 误区：改写越详细越好。
		- 过度改写会引入假设，反而偏离用户意图。
	- 误区：Query Rewriting 可以替代检索策略。
		- 它只是改善 query 表达，仍需要合适的切分、检索、融合和重排序。
