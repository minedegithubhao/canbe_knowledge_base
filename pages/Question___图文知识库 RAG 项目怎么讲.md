- **题目**
	- 图文知识库 RAG 项目怎么讲？

- **标准回答**
	- 这个项目的核心是图文证据管理，而不是简单把图片丢给大模型。很多知识在流程图、截图、表格和页面布局里，正文不一定有完整描述，所以要把图片 OCR、caption、图片 metadata、上下文页面、标题层级和附件来源统一管理起来。
	- 检索上不能只做文本向量检索，要同时考虑文本检索、图片检索、图文对齐检索和融合重排。生成时要基于文本和图片证据回答，并保留来源。隐私上，图片和附件要放在 `assets/`，发布前要检查 private 页面和私有附件，避免 GitHub Pages 泄露。

- **回答结构**
	- 场景：知识库里同时有文本、截图、流程图、架构图和白板。
	- 难点：图中信息不一定出现在正文里，OCR/caption 可能丢结构。
	- 入库：OCR、caption、metadata、来源、页面上下文。
	- 检索：文本检索、图片检索、图文融合、Rerank。
	- 生成：基于证据回答，保留来源，不支持时拒答。
	- 风险：隐私图片、附件、白板不应被 LLM 随意改写或公开。

- **项目追问**
	- OCR 和 caption 有什么区别？
		- OCR 提取图中文字，caption 描述图像语义；流程关系、空间结构和表格信息可能需要额外结构化。
	- 为什么不能只用全局图片向量？
		- 全局向量适合相似图检索，但细粒度问答还需要 OCR、caption、区域信息和页面上下文。
	- 白板怎么处理？
		- 白板适合人维护，LLM 默认不改 `whiteboards/*.edn`，可以把稳定结构整理成 Markdown 或 Mermaid。

- **我的易错点**
	- 不要把图文 RAG 讲成“图片转文字后做普通 RAG”。
	- 要讲证据融合和来源追踪。
	- 要主动提隐私和附件发布风险。
