- **项目目标**
	- 用 Python 手册作为知识库，构建一个可评估、可优化的手册类 RAG 系统。
	- 重点不是一次性调到最高分，而是通过评估日志定位问题，再逐步调整检索链路。

- **背景资料**
	- 原始项目笔记保留在 [[手册类RAG实战]]。
	- 手册内容结构化明显，有章节、编号、条目和解释。
	- 用户问题往往需要精确定位某个章节或概念。

- **系统架构**
	- 文档层：Python 手册、章节结构、编号、正文。
	- 入库层：清洗、切分、摘要或原文向量、metadata。
	- 检索层：向量检索、关键词检索、自查询检索。
	- 排序层：Rerank、阈值过滤、Top K 控制。
	- 评估层：RAGAS/PAGAS、检索日志、错误 case 复盘。

- **核心流程**
	- 初始评估
		- 部分 case 召回率高但准确率低，说明 Top K 中混入了垃圾信息。
		- 先判断问题在检索纯度，而不是直接修改 Prompt。
	- 引入 [[Concept/重排序]]
		- 对混合检索结果直接精排，并设置过滤阈值。
		- 效果是准确率和真实性提升，但部分有用 chunk 因评分低被过滤。
	- 调整 [[Concept/文档切分]]
		- 将 chunk size 调整到更能保留语义特征的粒度，并增加 overlap。
		- 结果是部分样本召回更稳定，但 chunk 变大后也可能降低纯度。
	- 尝试章节信息和摘要向量
		- 章节信息对某些 BM25 问题不一定有效，尤其受分词和编号格式影响。
		- 摘要向量能降噪，但不一定解决精确定位问题。
	- 引入 metadata/self-query 思路
		- 用结构化条件缩小检索范围，类似在 SQL 中增加过滤条件。
		- 准确率提升明显，但 Top K 和 Rerank 阈值仍需要继续权衡。
	- 调整 Top K
		- 提高 Top K 可以提高召回率，也可能引入噪声并降低真实性。

- **问题记录**
	- 召回率高但准确率低：说明正确信息可能在 Top K 中，但上下文噪声过多。
	- Rerank 后部分必要片段被过滤：说明 chunk 切分后关键特征不足，或阈值过高。
	- chunk 变大后关键词被稀释：说明语义完整性和关键词可检索性需要平衡。
	- Top K 增大后真实性可能下降：说明召回更多证据的同时也引入了噪声。

- **指标和验收**
	- 证据命中率：标准答案所需章节或 chunk 是否进入候选证据。
	- 证据纯度：Top K 中无关 chunk 占比是否下降。
	- 答案真实性：回答是否严格基于手册证据，不编造不存在的规则。
	- 答案相关性：回答是否真正回应用户问题，而不是只复述相邻章节。
	- 参数收益：chunk size、Top K、Rerank 阈值和 metadata 过滤的改动是否在评估集上带来稳定收益。

- **经验沉淀**
	- 先看评估日志，再决定改切分、检索、Rerank、Prompt 还是 Top K。
	- Rerank 能提高上下文纯度，但不能修复“正确证据没有被召回”的问题。
	- chunk size 不是越大越好，增大 chunk 可能提升语义完整性，也可能稀释关键词。
	- 摘要向量不是万能的，它能强化语义特征，但不能替代精确过滤。
	- metadata 过滤对手册类文档非常重要，尤其是章节、版本、模块和编号。

- **面试表达**
	- 这个项目最能体现什么能力？ #card
		- 它体现的是评估驱动优化能力。我不是直接调 Prompt，而是先用 RAGAS/PAGAS 和检索日志判断问题出在召回纯度、chunk 边界、Rerank 阈值还是 Top K，再逐步实验和复盘。
	- 为什么 Top K 提高后真实性可能下降？ #card
		- 因为 Top K 变大虽然可能召回更多相关证据，但也会带入更多噪声。噪声进入上下文后会干扰模型生成，导致答案不够忠实或不够简洁。
	- 召回率高但准确率低怎么处理？ #card
		- 先看正确证据是否在 Top K 中。如果在但混入噪声，优先考虑 Rerank、chunk 粒度、metadata 过滤和 Top K 控制，而不是直接改 Prompt。
	- 手册类 RAG 的验收指标怎么设计？ #card
		- 我会拆成检索和生成两段：检索看证据命中率和 Top K 纯度，生成看真实性、相关性和关键点覆盖；每次改 chunk、Top K 或 Rerank 阈值都要看收益和副作用。
