- **一句话定义**
	- 混合检索是把向量检索和关键词检索等多路召回结果合并，用互补方式提高 RAG 召回稳定性。

- **解决的问题**
	- 向量检索擅长语义相似，但对编号、专有名词、产品名、法条、函数名等精确匹配不稳定。
	- 关键词检索擅长字面匹配，但对口语化、同义表达和语义改写不稳定。
	- 单一路检索容易在某些问题上漏召回，混合检索可以提高鲁棒性。

- **典型流程**
	- 用户 query 先经过必要的 [[Concept/Query Rewriting]]。
	- 一路走向量检索，找语义相似 chunk。
	- 一路走关键词检索，例如 ES/BM25，找精确词和实体。
	- 对两路结果做去重、归一化、加权融合或 [[Concept/RRF]]。
	- 再交给 [[Concept/重排序]] 做精排。
	- 最后把 Top K 证据送入 Prompt。

- **什么时候特别需要**
	- 用户问题包含编号、章节、函数名、产品型号、专有名词。
	- 文档里有大量相似语义但关键条件不同的内容。
	- FAQ、法律、医疗、金融、客服、代码文档等对精确词敏感的场景。
	- 长篇小说或垂直领域文本中存在大量领域黑话和角色名。

- **工程注意点**
	- ES/BM25 的效果取决于分词器、字段设计、停用词、同义词和 metadata。
	- 如果 MySQL 是主库，ES 通常作为检索索引，不应该在召回阶段每条结果再回查 MySQL。
	- MySQL 到 ES 的一致性可通过 Binlog、Canal、消息队列和定时补偿任务维护。
	- 多路结果融合后仍需要看日志，避免一路召回大量低质量内容污染上下文。

- **面试表达**
	- 为什么要做混合检索？ #card
		- 因为向量检索和关键词检索互补。向量检索适合同义表达和语义相似，关键词检索适合编号、实体、术语和精确匹配。混合检索能降低单一路检索漏召回的风险。
	- 混合检索后为什么还需要 Rerank？ #card
		- 因为混合检索提高的是候选覆盖率，候选里仍然可能有噪声。Rerank 用更强的相关性模型筛出最适合放进 Prompt 的证据。
	- 混合检索为什么不能简单拼接结果？ #card
		- 因为不同召回方式的分数不可比，结果还可能重复或互相污染。需要去重、融合和精排，否则低质量召回可能挤占最终上下文。

- **常见误区**
	- 误区：混合检索等于简单拼接两个列表。
		- 实际需要去重、归一化、融合和精排，否则会被重复内容或低质量召回污染。
	- 误区：BM25 一定能解决关键词问题。
		- BM25 也受分词、编号格式、字段设计和 query 改写影响。
