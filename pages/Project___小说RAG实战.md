- **项目目标**
	- 面向长篇小说构建考据和写作助手，重点解决长文本、角色关系、剧情时序、专有名词和描写噪声。

- **背景资料**
	- 原始项目笔记保留在 [[小说RAG实战]]。
	- 项目背景和数据分析保留在 [[小说RAG实战数据分析报告]]。
	- 长篇小说篇幅很长，远超上下文窗口。
	- 人物、势力、法宝、事件和关系复杂。
	- 同一角色或设定会随章节推进变化。

- **系统架构**
	- 文档层：原文、章节、卷、位置、角色和事件。
	- 索引层：原文 chunk、章节摘要、角色档案、metadata。
	- 检索层：向量检索、BM25、Query Rewriting。
	- 排序层：Rerank、时间顺序、证据筛选。
	- 生成层：基于证据回答，并标注卷、章或来源片段。

- **核心流程**
	- 直接 Chunk Embedding
		- 把原文直接切片向量化。
		- 解决了“能不能搜到”的问题，但描写噪声多，检索容易跑偏。
	- Summary-based Indexing
		- 用模型把章节脱水成摘要，再对摘要向量化。
		- 作用是降噪和聚合，但不能替代精确检索。
	- [[Concept/混合检索]]
		- 向量检索负责语义，BM25 负责角色名、招式名、地名等精确词。
	- [[Concept/重排序]]
		- 对候选片段重新打分，优先保留事件、设定、动作和对话证据。
	- 时间序逻辑
		- 用章节号、卷、location_pct 等 metadata 保持因果顺序。
	- 引用标注
		- 要求答案标注卷、章或来源片段，降低随意编造。
	- [[Concept/Query Rewriting]]
		- 把口语问题改写成包含角色、时期、事件和专有名词的检索 query。

- **关键模块**
	- 章节 metadata。
	- 摘要索引。
	- 角色档案。
	- 关键词检索。
	- Rerank。
	- 引用标注。

- **问题记录**
	- 长程记忆
		- 普通 chunk 难以承载跨几百章的人物关系和伏笔，需要角色档案、世界观百科或 GraphRAG。
	- 时序一致性
		- 同一角色不同阶段状态不同，检索结果必须带时间顺序和章节范围。
	- 黑话与专有名词
		- 垂直领域词汇需要关键词检索、领域词表或 embedding 适配。
	- 上下文噪声
		- 描写、环境、无关对话会挤占上下文，需要摘要、Rerank 和证据筛选。

- **指标和验收**
	- 证据命中率：问题所需章节、角色、事件或设定是否被召回。
	- 时序一致性：回答是否符合章节顺序和角色阶段状态。
	- 专有名词准确率：角色名、法宝名、地点名和招式名是否正确。
	- 上下文信噪比：进入 Prompt 的候选是否减少无关描写和噪声。
	- 引用准确性：答案是否能标注卷、章或来源片段。
	- 重写漂移率：Query Rewriting 是否偏离原问题或引入不存在设定。

- **经验沉淀**
	- 小说 RAG 不能只按长度切 chunk，还要保留章节、卷、时间顺序和角色事件信息。
	- 专有名词需要关键词检索补充，不能只靠向量检索。
	- metadata 是防止时序错乱的基础。
	- GraphRAG 可以作为长程关系的扩展，但不是第一步。

- **面试表达**
	- 小说 RAG 和普通文档 RAG 有什么不同？ #card
		- 小说 RAG 的核心难点是长程记忆、时序一致性、角色关系和专有名词。普通文档更关注规则和事实定位，小说还要处理剧情演进和大量描写噪声。
	- 为什么小说 RAG 需要 metadata？ #card
		- 因为同一人物或设定会随章节变化。metadata 能记录卷、章、位置和时间顺序，防止模型把早期状态和后期状态混在一起。
	- 为什么要混合检索？ #card
		- 向量检索适合语义问题，但角色名、法宝名、招式名这类专有名词需要关键词检索补充，否则容易漏召回或召回偏移。
	- 为什么强模型做 Query Rewriting 反而可能变差？ #card
		- 重写任务需要稳定提取角色、地点、事件和专有名词。强模型如果加入文学化或推理化表达，可能超出 embedding 模型匹配能力，导致检索漂移。
