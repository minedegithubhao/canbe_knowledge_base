	- ## 📝 面试题（自问自答）
		- 角色关系的“长程记忆”问题
			- 痛点： 传统的 RAG 只是把文本切成碎片（Chunk）。当你问“主角和李长老是什么关系”时，如果检索出的片段只写了他们在一场战斗中对峙，模型可能误判他们是死敌，而忽略了前文 500 章铺垫的“师徒卧底”身份。
			- 解决方案： 构建知识图谱（GraphRAG）： 不要只存文本，要提取“人物-关系-事件”的实体链条。
			- 角色档案： 维护一个动态更新的“角色卡”，记录该角色的性格、修为、当前状态和核心人际关系。
		- 剧情的时序性与逻辑一致性
			- 痛点： 检索到的片段可能来自第一章，也可能来自第一千章。如果主角在第一章是废柴，第一千章是剑神，模型如果同时检索到这两个片段，就会出现逻辑混乱（幻觉）。
			- 解决方案： 时间戳/章节索引： 在 Metadata（元数据）中强制加入章节序号。
			- 滑动窗口摘要： 除了原始文本，额外存储每一卷或每十个章节的剧情梗概（Summary）。当用户提问时，先检索梗概，再定位细节。
		- 语义检索的“意境”与“黑话”误判，幻或修仙小说中充满大量的专有名词（如“筑基”、“道韵”、“天地法相”）。
			- 痛点： 通用的 Embedding 模型（如 OpenAI 的）可能无法理解这些玄幻黑话。比如“突破”在小说里是升级，在金融里是股价上涨。
			- 解决方案： * 微调（Fine-tuning）Embedding： 使用小说文本对向量模型进行微调，让它理解特定词汇的语义。
			- 混合检索： 必须结合 Keyword Search。当搜索特定招式名称（如“降龙十八掌”）时，关键词匹配比语义向量更可靠。
		- 上下文窗口与噪声干扰
			- 痛点： 检索出的 Top-K 片段中，可能含有大量的环境描写或无意义对话，挤占了模型极其有限的上下文窗口，导致核心剧情被淹没。
			- 解决方案： * 内容精简： 在喂给 LLM 之前，先用一个小模型或 Prompt 对检索到的片段进行“脱水”处理。
			- Rerank 策略： 针对小说逻辑进行重排序，优先保留含有“动作”、“对话”和“设定说明”的片段。
			  总结：你的开发优先级
		- 小说RAG优先级
			- 首要解决（基础）： 建立精准的章节元数据索引，防止时序错乱。
				- 正则切分，把小说按章节切开
					- 匹配规则示例：第[一二三\d]章
			- 核心突破（关键）： 引入 GraphRAG（知识图谱），处理复杂的人际关系和势力划分。
			- 体验优化（进阶）： 建立“世界观百科”和“角色档案”，作为 RAG 的常驻补充。
