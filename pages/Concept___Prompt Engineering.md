- **一句话定义**
	- Prompt Engineering 是通过设计指令、上下文、示例、输出格式和约束，让大模型更稳定完成目标任务的方法。

- **解决的问题**
	- 用户自然语言需求不够清楚，模型容易误解。
	- 任务要求复杂，模型一次性完成容易漏步骤。
	- 输出格式不稳定，难以被程序消费。
	- RAG 和 Agent 场景需要模型严格遵守证据、工具和边界。

- **核心原则**
	- 指令明确：说明角色、任务、背景、约束和输出格式。
	- 信息分隔：用清晰边界区分指令、上下文、用户输入和示例。
	- 示例引导：用少量示例稳定输出风格和结构。
	- 拆分任务：复杂任务拆成多个子任务或 [[Concept/Prompt Chaining]]。
	- 约束边界：明确不能编造、不能越权、证据不足时拒答。

- **项目中怎么用**
	- RAG 项目中，Prompt 要要求模型只基于检索证据回答，并在证据不足时拒答。
	- 评估集生成中，Prompt 要明确题目类型、目标组件、标准答案和禁止幻觉。
	- 面试刷题中，Prompt 可以让模型扮演面试官，但仍需要人工复核。

- **面试表达**
	- Prompt Engineering 的核心原则是什么？ #card
		- 核心是把任务说清楚、把上下文分隔清楚、给出输出格式和约束，复杂任务拆步骤，必要时提供示例，并明确证据不足时的处理方式。

- **常见误区**
	- 误区：Prompt 可以解决所有大模型问题。
		- Prompt 只能控制生成阶段；如果数据、检索、工具或权限设计有问题，Prompt 无法稳定修复。
	- 误区：越长越详细越好。
		- 过长 Prompt 会增加噪声和成本，关键是结构清晰、约束有效。
